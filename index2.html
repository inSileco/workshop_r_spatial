R As A GIS

Why Use R As A GIS?

  • one environment for import, cleaning, spatial processing, modeling, and reporting
  • reproducible workflows instead of manual click sequences
  • transparent transformations and analysis choices
  • easy transition from exploration to automation

Spatial Workflow Anatomy

  1. import data
  2. create or validate geometry
  3. inspect CRS and transform if needed
  4. restrict to a study area
  5. measure, join, intersect, extract
  6. summarise for analysis
  7. model, map, and export

Spatial Data At A Macro Level

Vector

  • discrete features
  • geometry + attributes
  • points, lines, polygons
  • typical questions:
    • where?
    • how far?
    • what overlaps what?

Raster

  • gridded surfaces
  • cells with values
  • extent, resolution, CRS
  • typical questions:
    • what value is here?
    • how does the surface vary?
    • what is the average over an area?

Main Packages

sf

  • vector data
  • simple features standard
  • geometry-aware data frames

stars

  • raster and arrays
  • good fit with sf
  • convenient for plotting and extraction

terra

  • vector + raster in one package
  • strong for raster operations
  • consistent GIS-style workflow

Supporting Packages

  • dplyr for data manipulation
  • units for explicit measurement handling in sf
  • mapview for rapid interactive mapping
  • ggplot2 for polished static figures
  • MASS::kde2d() for simple KDE examples

Vectors

Why Vectors First?

  • vectors are usually more intuitive for beginners
  • both workshop storylines start from tabular coordinates
  • joins, intersections, and measurements are easier to explain here first
  • raster integration makes more sense once the vector workflow is stable

The Vector Data Model

  • features have attributes and geometry
  • geometry lives in a CRS
  • geometry types matter:
    • points for observations
    • lines for trajectories
    • polygons for regions and habitats

Two Vector Storylines

Points route

  • bird observations
  • start from tabular lon/lat
  • create points
  • join to polygon context

Lines route

  • telemetry fixes
  • start from lon/lat + timestamp
  • create points
  • rebuild tracks from ordered points

Import Vector And Tabular Data

sf

birds <- read.csv("bird_observations.csv")
ecodistricts <- st_read("ecodistricts.gpkg")

terra

birds <- read.csv("bird_observations.csv")
ecodistricts <- vect("ecodistricts.gpkg")
  • key idea: import attributes and import geometry are separate concerns

Create Point Geometry From lon/lat

sf

sites <- birds |>
  st_as_sf(coords = c("longitude", "latitude"),
           crs = 4326,
           remove = FALSE)

terra

sites <- vect(birds,
              geom = c("longitude", "latitude"),
              crs = "EPSG:4326")

Rebuild Lines From Ordered Points

  • start from telemetry points
  • sort by individual and timestamp
  • group ordered coordinates into tracks
  • line building is not a separate data source
  • it is a geometry-construction step

This is an important pedagogic distinction: participants are not “reading lines”, they are constructing them.

CRS: What We Need Operationally

  • inspect current CRS
  • know whether data are geographic or projected
  • choose a working CRS for measurement and analysis
  • transform all relevant layers consistently

CRS API Lens

sf

st_crs(sites)
sites_qc <- st_transform(sites, 32198)

terra

crs(sites)
sites_qc <- project(sites, "EPSG:32198")

Restrict To The Study Area

  • keep only the relevant geographic scope
  • reduce clutter and computation
  • make later extraction and summaries easier
  • for vectors this is usually:
    • filter
    • crop
    • keep overlapping features

Study-Area Restriction API Lens

sf

sites <- st_filter(sites, east)
ecodistricts <- st_filter(ecodistricts, east)

terra

sites <- crop(sites, east)
ecodistricts <- crop(ecodistricts, east)

Vector Measurements

  • polygon area
  • line length
  • distance from points or tracks to coast
  • measurement depends on CRS

Note

sf often makes units explicit through units. terra usually returns numeric values whose interpretation depends on CRS and function behavior.

Measurement API Lens

sf

ecodistricts$area <- st_area(ecodistricts)
sites$coast_distance <- st_distance(sites, coast)

terra

ecodistricts$area_km2 <- expanse(ecodistricts, unit = "km")
sites$coast_distance_km <- distance(sites, coast)[, 1] / 1000

Vector Relationships

Join logic

  • attach polygon attributes to points
  • ask which polygon contains each feature
  • enrich observations with context

Intersection logic

  • create new geometries from overlap
  • for line workflows, create habitat-attributed track segments
  • this is not only an attribute join

Relationship API Lens

sf

sites <- st_join(sites, ecodistricts)
track_segments <- st_intersection(tracks, habitats)

terra

site_values <- extract(ecodistricts, sites)
track_segments <- intersect(tracks, habitats)

Quick Vector Mapping

  • plot() for fast sanity checks
  • mapview() for interactive exploration
  • the first map is often a QA/QC step, not a publication figure

Day 2

Rasters

The Raster Data Model

  • a raster is a grid of cells with values
  • key metadata:
    • extent
    • resolution
    • CRS
    • number of layers
  • raster operations often modify the grid itself, not only attributes

Why Rasters Need Their Own Section

  • different mental model than vectors
  • projection can imply resampling
  • crop, mask, and extraction have different meanings than with vectors
  • learners benefit from seeing this as a separate object system first

Import And Inspect Rasters

stars

temperature <- read_stars("temperature.tif")
plot(temperature)

terra

temperature <- rast("temperature.tif")
plot(temperature)

What To Inspect Before Doing Anything Else

  • CRS
  • layer names
  • extent
  • resolution
  • whether the raster already covers the study area

Crop, Mask, Project

Crop

  • reduce extent
  • keep a rectangular window or study boundary extent

Mask

  • keep values only inside a geometry
  • remove cells outside a polygon

Project

  • change CRS
  • usually implies resampling
  • use carefully

Raster Transformation API Lens

sf/stars

bathymetry <- st_warp(bathymetry, crs = st_crs(tracks))
bathymetry <- st_crop(bathymetry, st_bbox(tracks))

terra

bathymetry <- project(bathymetry, target_crs)
bathymetry <- crop(bathymetry, ext(tracks))

Quick Raster Mapping

  • use plot() to inspect values and spatial coverage
  • use mapview() for interactive exploration where helpful
  • use geom_stars() later when building a final static map

Integrated Workflows

Why Integration Matters

  • most analyses combine feature geometry with gridded context
  • vectors define where
  • rasters contribute environmental values
  • analysis tables sit at the interface between the two

Vector-Raster Extraction

  • points: extract cell values at observation locations
  • lines: extract summaries along tracks or segments
  • segments: connect movement geometry to environmental context

Extraction API Lens

sf/stars

sites$tavg_january <- st_extract(temperature, sites)[[1]][, 1]
segment_bathymetry <- st_extract(bathymetry, track_segments, FUN = mean)

terra

temperature_values <- extract(temperature, sites)
segment_bathymetry <- extract(bathymetry, track_segments, fun = mean)

Build Analysis Tables

  • keep only the attributes needed for modeling
  • drop geometry when the spatial step is complete
  • summarise by polygon, track, or segment
  • compute grouped metrics for later analysis

Table-Building Verbs

  • st_drop_geometry() or as.data.frame()
  • select()
  • mutate()
  • group_by()
  • summarise()
  • count()
  • left_join()

Analysis Menu In This Workshop

Core

Advanced

Simple Aggregated Modeling

  • points route:
    • counts by ecodistrict
    • simple Poisson GLM
  • lines route:
    • summaries by track or segment
    • model-ready tables from overlap and extraction

Advanced Analysis: Pseudo-Absence GLM

  • points workflow only
  • sample pseudo-absences
  • extract the same predictors as for presences
  • fit a simple binomial GLM

Advanced Analysis: Segment-Level GLM

  • lines workflow only
  • use habitat-attributed segments
  • integrate segment length, environmental summaries, and habitat context
  • fit a Gamma GLM on segment-level outcomes

Advanced Analysis: KDE

  • available in both storylines
  • used to create a smooth surface of concentration or intensity
  • useful for exploratory visualization and discussion

Note

In the workshop scripts, KDE values are scaled to 0-1 for plotting convenience only. The raw density concept is unchanged.
